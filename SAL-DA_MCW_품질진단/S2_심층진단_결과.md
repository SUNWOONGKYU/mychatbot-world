## AP (API 설계) — 4카드

### S2AP1: 엔드포인트 명명 일관성 — Warning / Medium
- Issues: create-bot(동사) vs bots(명사) 이중 경로 존재
- wiki 서브시스템 전체가 RPC 스타일(accumulate, ingest, lint, query) — 다른 섹션(REST)과 충돌
- jobs/hire, jobs/settle 등 동사형 서브경로 혼재
- Impact: 외부 연동 또는 추가 개발 시 경로 규칙 파악 불가, API 문서화 비용 증가
- Improvement Plan: REST 명사 기반으로 통일 (create-bot → POST /api/bots), wiki RPC는 /api/wiki/[action] 네임스페이스 유지하되 문서화

### S2AP2: 에러 응답 형식 통일 — Warning / High
- Issues: 패턴 A: `{ success: false, error: "...", data: null }` (create-bot, bots, kb 계열)
- 패턴 B: `{ error: "..." }` (community, jobs, skills 계열)
- 에러 메시지 언어 혼재 (한국어 + 영어)
- 프론트엔드 에러 처리 분기 복잡도 증가
- Impact: 프론트엔드 에러 핸들링 분기 이중화, 언어 혼재로 사용자 노출 에러 메시지 불일치
- Improvement Plan: 공통 에러 응답 타입 정의 (lib/api-response.ts), 모든 route에 통일 형식 적용, 메시지 한국어 통일

### S2AP3: 입력 검증 — Warning / Medium
- Issues: Zod/Yup/Joi 등 스키마 검증 라이브러리 **0건** (완전 미사용)
- 수동 if 체크 일관성 없음 (create-bot은 철저, wiki/ingest는 느슨)
- `req.json().catch(() => ({}))` 패턴 — JSON 파싱 실패를 무시
- Impact: 잘못된 입력값이 DB에 삽입되거나 서버 오류 유발, 잘못된 JSON 요청 무시로 디버깅 불가
- Improvement Plan: Zod 도입 후 route별 입력 스키마 정의, req.json() catch 시 400 응답 반환으로 통일

### S2AP4: HTTP 메서드 적절성 — Normal / Low
- Issues: GET으로 데이터 변경 0건 (양호)
- PATCH만 사용 (PUT 미사용) — 부분 업데이트에 적합
- DELETE /api/bots?id=xxx (쿼리스트링) — 동적 경로 미사용이나 기능상 문제 없음
- Impact: 낮음. 현재 방식으로 기능상 문제 없음. DELETE 경로 스타일만 비표준
- Improvement Plan: 현 패턴 유지. DELETE는 점진적으로 /api/bots/[id] 동적 경로로 전환 검토

## MT (유지보수성) — 5카드

### S2MT1: 코드 중복도 — Warning / High
- Issues: `createClient(url, key)` 초기화 코드 **21개 route에 반복**
- lib/supabase.ts 싱글턴이 존재하나 app/ 전체에서 2곳만 import
- 환경변수 키 이름 변경 시 21곳 수동 수정 필요
- Impact: 환경변수 키 변경 또는 Supabase 클라이언트 버전 업 시 21곳 수동 수정, 누락 버그 확률 높음
- Improvement Plan: lib/supabase.ts 싱글턴 함수를 모든 route에서 import 통일. createClient 직접 호출 전면 제거

### S2MT2: 타입 안전성 — **Danger** / **Critical**
- Issues: `: any` 사용 **116회**, `as any` 캐스팅 **112회** = 총 228회
- `createClient(url, key) as any` — Supabase 클라이언트 타입 완전 손실
- types/ 폴더의 공통 타입이 app/에서 사실상 미참조
- Impact: 타입 오류가 컴파일 단계에서 감지 안 됨 → 런타임 크래시. TypeScript 도입 효과 사실상 없음
- Improvement Plan: Supabase codegen으로 DB 타입 자동 생성, types/ 공통 타입 app/ 전체 적용. any 점진적 제거 (eslint no-explicit-any 도입)

### S2MT3: 에러 경계 — **Danger** / **Critical**
- Issues: error.tsx, not-found.tsx, loading.tsx **전부 없음** (app/ 전체)
- ErrorBoundary 컴포넌트 없음
- 렌더링 에러 → Next.js 기본 흰 화면 직행
- Impact: 사용자가 에러 발생 시 흰 화면만 보임. 페이지 이탈률 급등, 브랜드 신뢰도 손상
- Improvement Plan: app/error.tsx, app/not-found.tsx, app/loading.tsx 생성. 주요 페이지별 세분화 error boundary 추가

### S2MT4: 레거시 코드 정리도 — Warning / Medium
- Issues: js/ 폴더: Vanilla JS 22개 파일 잔존
- pages/: Vanilla HTML 구조 잔존 (Next.js Pages Router 아님)
- api/: 루트 레벨 Vanilla API 잔존 (app/api/와 중복 가능성)
- Impact: 번들에 불필요한 코드 포함, app/api/와 중복 route 충돌 가능성, 신규 개발자 혼선
- Improvement Plan: js/ 폴더 기능 이전 후 삭제. pages/ 및 루트 api/ 는 app/ 대응 route 확인 후 제거

### S2MT5: CSS 아키텍처 일관성 — Warning / Medium
- Issues: 3중 혼재: Tailwind(1,459회) + css/ 커스텀(10,752줄) + 인라인 style(1,076건)
- css/ 파일 3개(home.css, learning.css, styles.css)가 layout.tsx에 미import → 데드코드?
- 인라인 style 1,076건 → 디자인 토큰 변경 시 대규모 수정 필요
- Impact: 디자인 시스템 변경 시 1,076곳 수동 수정, 미사용 CSS 파일로 번들 사이즈 증가
- Improvement Plan: css/ 데드코드 파일 식별 후 제거, 인라인 style을 Tailwind 클래스로 점진 교체, CSS 변수 기반 디자인 토큰 도입
