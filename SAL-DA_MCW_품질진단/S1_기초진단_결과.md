## SE (보안) — 5카드

### S1SE1: API 인증 미들웨어 부재 — Warning / Medium
- Issues: middleware.ts 없음. 인증 로직 3가지 패턴 분산 (getSession, getUser, X-Admin-Key)
- lib/admin-auth.ts의 requireAdmin()이 admin route에서 미사용 (로컬 중복 정의)
- Impact: 새 route 추가 시 인증 누락 가능, 인증 버그 수정 시 3곳 모두 전파 실패
- Improvement Plan: middleware.ts 생성하여 /api/* 경로 중앙 인증 게이트 구축, requireAdmin() 통합

### S1SE2: API Route 권한 검증 일관성 — Warning / **Critical**
- Issues: `/api/ai/chat` 완전 무인증 — 누구나 OpenRouter AI 호출 가능 (비용 폭탄)
- `/api/wiki/ingest` 인증 없이 service_role로 DB 쓰기 가능
- Impact: 악의적 사용자가 AI API를 무제한 호출 → 월 수십만 원 비용 피해, 비인가 DB 쓰기 가능
- Improvement Plan: 각 route에 Bearer 토큰 인증 추가, /api/wiki/ingest에 관리자 권한 검증 적용

### S1SE3: 환경변수 노출 — **Danger** / **Critical**
- Issues: `.env`에 실제 시크릿 키 평문 존재 (OPENROUTER_API_KEY, SUPABASE_SERVICE_ROLE_KEY 등)
- 결제 계좌정보 하드코딩 폴백 (payments/route.ts:236-238)
- getSession() 서버사이드 사용 (Supabase 공식 비권장)
- Impact: .env 파일 유출 시 DB, AI API, 결제 계좌 전체 접근 가능. 계좌 오기재로 결제 오류 유발
- Improvement Plan: .env를 .gitignore 확인, 계좌 하드코딩 폴백 제거 후 환경변수 필수화, getSession() → getUser()로 교체

### S1SE4: XSS/인젝션 방어 — Warning / High
- Issues: dangerouslySetInnerHTML 3곳 (chat-window, Step8Deploy, wizard-steps) — sanitize 없음
- jobs search: ilike 쿼리에 검색어 직접 문자열 보간
- Impact: 악성 HTML/스크립트 삽입 시 사용자 세션 탈취, 피싱 페이지 렌더링, DB 데이터 노출 가능
- Improvement Plan: DOMPurify 또는 sanitize-html 적용, ilike 쿼리 파라미터화 처리

### S1SE5: Supabase RLS 정책 — Warning / High
- Issues: 대부분 API route가 service_role 키 사용 → RLS 사실상 미동작
- 코드 레벨 .eq('user_id') 필터에 의존 → 누락 시 즉시 데이터 유출
- Impact: 코드 버그 1개로 전체 사용자 데이터 무단 접근 가능, 보안 레이어 단일화로 방어 깊이 없음
- Improvement Plan: 공개 데이터 제외 route에서 anon key + RLS 활용 전환, service_role 사용 범위 최소화

## DB (데이터베이스) — 3카드

### S1DB1: 스키마 정규화 수준 — Warning / High
- Issues: personas vs mcw_personas 테이블 중복 존재 (역할 불명확)
- bots_to_inherit UUID[] vs mcw_bots.id TEXT 타입 불일치 → 런타임 오류
- chatbot_memory, chatbot_search_history, bot_growth, bot_job_applications 4개 테이블 마이그레이션 미등록
- mcw_bots.faqs JSONB 컬럼과 faqs 테이블 중복
- Impact: 타입 불일치로 상속 기능 런타임 오류, 마이그레이션 미등록으로 재배포 시 스키마 누락, 중복 테이블로 데이터 불일치 발생
- Improvement Plan: personas/mcw_personas 역할 정의 후 통합, 타입 일치 마이그레이션 작성, 4개 테이블 migration 파일 등록

### S1DB2: 쿼리 효율성 — Warning / High
- Issues: wiki/ingest: 루프 내 N+1 쿼리 (KB 아이템당 SELECT)
- sync/route.ts: 루프 내 순차 HTTP 호출 (KB당 /api/kb/embed)
- select('*') 21곳 — embedding vector(1536) 포함 테이블에서 불필요한 데이터 전송
- admin/users: 전체 사용자 목록 메모리 로드 후 필터
- Impact: 대용량 KB 처리 시 응답 시간 수십 초, 네트워크 대역폭 낭비, Supabase 요청 수 폭증으로 과금 증가
- Improvement Plan: N+1 → 배치 쿼리 전환, select('*') → 필요 컬럼 명시, admin/users 서버사이드 페이지네이션 적용

### S1DB3: RLS 정책 완전성 — **Danger** / **Critical**
- Issues: 서버 API Route 12개에서 anon key 사용 + getSession() → 비결정적 동작
- match_kb_items RPC: anon에게 EXECUTE 권한 부여 → 비인증 사용자 KB 검색 가능
- mcw_inheritance_event_logs INSERT 정책 없음
- 초기 마이그레이션 USING(true) 정책이 교체되었는지 불확실
- Impact: 비인증 사용자가 타인의 KB 데이터 검색 가능, INSERT 정책 공백으로 이벤트 로그 무단 삽입 가능
- Improvement Plan: anon key route를 service_role로 전환하거나 RLS 정책 보완, match_kb_items TO anon 권한 회수, INSERT 정책 추가
