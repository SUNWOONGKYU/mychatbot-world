# MCW 관리자 대시보드 기능 전수조사

조사일: 2026-04-07
조사 범위: Vanilla 원본 + React 전환본 + API + DB 스키마 전체

---

## 조사 결론 요약

MCW 관리자 대시보드는 **Vanilla 원본** (`pages/admin/index.html`) 기준으로 5개 섹션이 구현되어 있고, **React 전용 admin 디렉토리는 존재하지 않는다** (`app/admin/` 없음). 관리자 API는 `app/api/admin/` 하위에 5개 엔드포인트가 있으며, 수익·정산 관련 기능은 일반 사용자용 API로 구현되어 있다 (관리자 전용 정산 처리 API 미구현).

## 1. 회원 관리

### 기능 목록

| 기능 | 설명 |
|------|------|
| 사용자 목록 조회 | 페이지네이션 (20건/페이지, 최대 100) |
| 이메일/이름 검색 | 검색어로 필터링 |
| 사용자 정보 표시 | 이메일, 이름, 가입일, 크레딧 잔액 |
| 크레딧 잔액 조회 | mcw_credits 테이블과 auth.users 조인 |

### 구현 현황

- **Vanilla**: `pages/admin/index.html` — L113~L121 (사용자 관리 섹션), L234~L249 (loadUsers 함수)
- **React**: 미구현 (app/admin/ 디렉토리 없음)
- **API**: `app/api/admin/users/route.ts` — GET /api/admin/users
  - 쿼리: search, page, limit
  - 응답: { users, total, page, totalPages }
  - users 필드: id, email, displayName, createdAt, lastSignInAt, balance, totalPurchased

### DB 테이블

- `auth.users` (Supabase 내장) — 사용자 인증 정보
- `mcw_credits` — 크레딧 잔액 (user_id, balance, total_purchased, total_used, total_refunded, total_bonus)

### 미구현 기능 (계획만 있음)

- 사용자 상세 보기 (클릭 시 모달/페이지)
- 크레딧 직접 조정 (admin 부여/차감)
- 사용자 정지/밴 처리
- 사용자 상세 활동 이력 조회

## 2. 입금/결제 관리

### 기능 목록

| 기능 | 설명 |
|------|------|
| 결제 목록 조회 | 상태 필터(pending/completed/all) + 페이지네이션 |
| 입금 대기 목록 | 대기 중 입금 신청 목록 |
| 입금 승인 | 상태 completed 변경 + 크레딧 충전 + 이력 기록 |
| 입금 거부 | 상태 cancelled 변경 |
| 대시보드 미리보기 | 최근 pending 5건 바로 승인/거부 가능 |

### 구현 현황

- **Vanilla**: `pages/admin/index.html`
  - L124~L134 (결제 관리 섹션 HTML)
  - L252~L298 (loadPayments, fetchPayments, approvePayment, rejectPayment 함수)
- **React**: 미구현
- **API (관리자용)**: `app/api/admin/payments/route.ts`
  - GET /api/admin/payments — status 필터, 페이지네이션
  - PATCH /api/admin/payments — approve/reject 액션
- **API (일반 사용자용)**: `app/api/payments/route.ts`
  - GET /api/payments — 본인 결제 내역
  - POST /api/payments — 무통장 입금 요청 생성
- **API (confirm)**: `app/api/payments/confirm/route.ts`
  - PATCH /api/payments/confirm — 관리자 입금 확인 (X-Admin-Key 인증)

### 승인 처리 흐름

```
PATCH /api/admin/payments { action: 'approve' }
  → mcw_payments.status = 'completed'
  → mcw_credits.balance += amount (upsert)
  → mcw_credit_transactions INSERT (type: 'purchase')
```

### DB 테이블

- `mcw_payments` — 결제 레코드
  - id, user_id, provider, amount, credit_amount, status
  - payment_type, bank_name, account_number, account_holder
  - confirmed_at, confirmed_by (관리자 승인 시 기록)
  - 허용 상태: pending / completed / failed / cancelled / refunded
- `mcw_credits` — 크레딧 잔액
- `mcw_credit_transactions` — 크레딧 충전/차감 이력
  - type: purchase / use / refund / bonus / admin

### 충전 가능 금액

1,000 / 5,000 / 10,000 / 50,000 (KRW, 1 크레딧 = 1원)

### 미구현 기능

- 결제 상세 정보 모달 (입금자명, 계좌 정보, 처리 히스토리)
- 환불 처리 (refunded 상태)
- 일괄 승인 기능
- 입금 신청자에게 알림 발송

## 5. 수익/정산 관리

이 기능은 관리자 전용이 아니라 **봇 소유자(크리에이터)** 용으로 구현되어 있다. 관리자는 정산 요청을 승인하는 역할이지만, **관리자용 정산 승인 API는 미구현** 상태다.

### 기능 목록

| 기능 | 대상 | 설명 |
|------|------|------|
| 비즈니스 대시보드 | 크리에이터 | 총수익, 이번 달 수익, 미정산 금액 |
| 월별 수익 추이 | 크리에이터 | Canvas 바 차트, 6개월 |
| 봇별 수익 비율 | 크리에이터 | 봇별 수익 바 차트 |
| 매출 상세 조회 | 크리에이터 | 날짜 필터, 소스 필터, 일별/월별 집계 |
| 정산 요청 | 크리에이터 | 계좌 정보 입력 후 정산 신청 |
| 정산 내역 조회 | 크리에이터 | 상태별 필터 (pending/processing/completed/failed) |

### 구현 현황

- **React 페이지**:
  - `app/business/page.tsx` + `app/business/page-client.tsx` — 비즈니스 대시보드
  - `app/business/revenue/page.tsx` — 매출 상세 (날짜/소스 필터, 일별 목록)
  - `app/business/settlement/page.tsx` — 정산 관리 (요청, 이력, 결제수단)
- **API**:
  - `app/api/revenue/route.ts` — GET /api/revenue (daily/monthly 집계)
  - `app/api/revenue/settlements/route.ts`
    - GET /api/revenue/settlements — 정산 내역
    - POST /api/revenue/settlements — 정산 요청 (최소 10,000원)

### DB 테이블

- `mcw_revenue` — 크리에이터 수익 레코드
  - creator_id, bot_id, type (bot_usage/template_purchase/subscription)
  - gross_amount, platform_fee, net_amount, settled, settlement_id
- `mcw_settlements` — 정산 요청 레코드
  - creator_id, status (pending/processing/completed/failed)
  - requested_amount, approved_amount, bank_name, bank_account_number
  - period_from, period_to, admin_note, fail_reason

### 미구현 기능 (관리자 측)

- 정산 요청 목록 조회 (관리자 뷰)
- 정산 승인/거부 API
- 정산 처리 후 settled 플래그 업데이트
- 관리자 노트 입력

## 8. 시스템/권한 관리

### DB 테이블 (존재하나 UI 없음)

- `admin_roles` — 관리자 역할 테이블
  - role: super_admin / admin / moderator
  - permissions: JSONB (세부 권한)
  - granted_by, granted_at, revoked_at
- `admin_audit_logs` — 관리자 감사 로그 (불변)
  - admin_id, action, target_type, target_id, details, ip_address

### 인증 방식

현재 관리자 인증은 두 가지 방식이 혼재:
1. **Vanilla**: `ADMIN_EMAILS` 배열로 이메일 하드코딩 (`pages/admin/index.html` L161)
2. **React API**: `X-Admin-Key` 헤더 + `ADMIN_API_KEY` 환경변수 비교

### 미구현 기능

- 관리자 역할 부여/회수 UI
- 감사 로그 조회 UI
- 다중 관리자 지원 (현재 단일 이메일만)
- IP 기반 접근 제한

## 주요 발견 사항

1. **React admin 페이지 없음**: `app/admin/` 디렉토리가 존재하지 않는다. React 전환 작업에서 관리자 대시보드는 아직 React로 이식되지 않았다.

2. **인증 방식 이원화**: Vanilla는 이메일 하드코딩, React API는 X-Admin-Key 헤더 방식. 통합 필요.

3. **정산 관리자 기능 공백**: 크리에이터가 정산 요청을 할 수 있는 API는 완성되어 있으나, 관리자가 이를 승인/처리하는 API와 UI가 전혀 없다. `mcw_settlements` 테이블의 `admin_note`, `approved_amount`, `processed_at` 컬럼은 있으나 업데이트 로직이 없다.

4. **admin_roles, admin_audit_logs 사용 안 함**: DB 테이블은 아카이브 마이그레이션에 정의되어 있으나 현재 운영 마이그레이션(`supabase/migrations/`)에 포함되지 않았고, 사용하는 코드도 없다.

5. **봇 신고 테이블 존재**: `bot_reports` 테이블이 `20260402_community_tables.sql`에 있으나, 관리자 처리 API와 UI가 없다.
